## 名词表

下面是全 skill（含各 reference）会用到的名词。出现这些词时按此处定义理解，不在正文反复解释。

| 名词 | 含义 |
|------|------|
| **run** | python-docx / Word 里**一段共享同一组字符格式（字体/字号/粗体/颜色等）的文本**，对应 Word XML 的 `<w:r>` 节点。一句话被上标/加粗/变色切成多个 run 很常见（详见 `references/doc-report.md`）。 |
| **single source of truth（单一可信源）** | 一份资产只在一处维护，其他地方只**引用、不复制**。本 skill 的图片资产据此只放项目级 `images/` 一处。 |
| **一手来源 / 二手来源** | 一手 = 发布主体的原始出处（政府门户、统计局原文、报告原件、正式文本）；二手 = 转述汇编（行业白皮书、新闻转述、百科）。数据与事实必须追到一手。 |
| **U / Q（色标字母）** | 降重用。**U** = Unquoted，查重报告中的【红】（未引证，计入"去除引用复制比"）；**Q** = Quote，【绿】（已标脚注，不计入"去除引用复制比"）。详见 `references/thesis.md`。 |
| **顺序编码制** | GB/T 7714-2015 的一种引用制度：正文引用处标**上标方括号序号** `[N]`，文末参考文献**按正文出现先后**编号列出。 |
| **占位符工作流** | 写作时用人类可读 key（如 `⟦han2025⟧`）占位，落版时一键替换成正确的 `[N]`；增删文献时只动 key，编号全自动重排。 |
| **codex-image** | 调 codex 让它用**图像生成模型真正"画"图**（而非退化成写 SVG/matplotlib 代码拼图）。 |
| **期刊分级标识** | 判断文献等级的收录/分级标记：CSSCI（南大核心）、北大核心、CSCD、EI、SCI、AMI、CSTPCD 等（同一刊常多个并列）。 |
| **F0–F4** | 模式 A 降重时的分章文件简称（`00_前言与参考文献.md` … `04_…`），定义见 `references/thesis.md`。 |
| **GFM** | GitHub Flavored Markdown，CommonMark 的扩展（管道表格 `| a | b |`、删除线、任务列表、代码块语言标注等）。 |

## 项目资产管理（跨模式，先定调）

**这是任何文档项目开工前要先定的基调。** docx、pptx、论文、其他 deliverable **共用一份图片资产与一份源文献**，不要每个产出物各自复制一份。

### single source of truth 原则

- 图片资产**只在项目级 `images/` 一处维护**；docx 内嵌、pptx 内嵌、SVG 引用、网页插图 —— **只引用、不复制**。
- 文件名带版本号（`v2`、`v3`），新版用**新文件名**不覆盖旧版，方便回滚。
- 每次产出物更新前，**用脚本从 `images/` 自动刷新内嵌图**（而非手动），避免"docx 用着 v1、PPT 已经 v2"的不一致。
- `images/` 是各产出物共用 —— **不要叫 `ppt/images/` 或 `docx/images/`**，那会绑定到某一类产出物。

### 推荐目录结构

```
<project>/
├── images/                      ← 通用图片资产（single source of truth）
│   ├── chart-<主题A>-v2.png      ← 数据图，带版本号
│   ├── chart-<主题B>-v2.png
│   ├── scene-<主题C>.png         ← 实景 / 概念插画
│   └── ...
├── mineru/                      ← MinerU 解析产物，与 images/ 平级（解析细节见 software skill 的 mineru.md）
│   └── <文献名>/                 ← 每篇 PDF 一个子目录：把 MinerU 结果 zip 原样解压到这里
│       ├── <原始>.pdf            ← 原始 PDF 一并留存
│       ├── full.md              ← 解析出的正文 Markdown
│       ├── *_content_list.json  ← md 块 ↔ 页码/坐标 的对应关系 JSON
│       └── images/              ← 该文献抽出的图（MinerU 自带，独立于项目级 images/）
├── sources/                     ← 一手原文 + 项目信息源/决策记录（含 PPT 时与 ppt-master 的 sources/ 同一目录）
│   ├── source-of-truth.md       ← 最高信息源声明 + 口述决策记录（持久化，仿 qcfd2026-site 的 SOURCES.md）
│   ├── reference.md             ← 引用关系总表（[N] → 文献 → 数据点）+ 逐数据点核验记录（原数据核验底稿并入）
│   └── *.pdf / *.html           ← 政策原文 / 未过 MinerU 的文献原文
└── output/                      ← 所有产出物（thesis 分章 md/回填稿、docx、pptx 都放这里）
    ├── <论文名>/                 ← thesis 分章 md + 回填 docx/pdf
    ├── <docx 短文>.docx          ← 内嵌图来自 images/
    └── <pptx 汇报>.pptx          ← 内嵌图来自 images/
```

### MinerU 解析产物：mineru/（与 images/ 平级）

- MinerU 把 PDF 转 md 的**结果 zip 统一原样解压到 `<project>/mineru/<文献名>/`**，与 `images/` 平级；原始 PDF 一并留在该子目录便于回查。
- zip 内含 `full.md`（正文）、对应关系 JSON（md 块 ↔ 页码/坐标）、`images/`（该文献抽出的图）等，是**只读的一手转录底稿** —— 引用核查/二次核验时回查用，不要直接当论文正文改。
- `mineru/<文献名>/images/` 是文献自带抽出图，**与项目级 `images/`（最终配图单一源）是两码事**，不要混。
- 解析参数、token、大文件拆分、加密 PDF 等操作细节见 `software` skill 的 `references/mineru.md`。

### 信息源与决策记录：sources/（持久化）

`sources/` 除了放一手原文（政策原文、未过 MinerU 的 pdf/html；MinerU 解析过的在 `mineru/`），还持久化两份**项目级真相底稿**，文件名一律小写：

- **`source-of-truth.md`** —— 项目「最高信息源 + 口述决策」总账（仿 1810 上 qcfd2026-site 的 `SOURCES.md`）。把"为什么这么写、依据谁、谁拍的板"落到文件、不靠对话记忆，换个 AI 或隔几周回来都能对齐。至少记四样：
  - **主源 / 历史源声明**：哪份文件、哪个版本是当前权威源；哪些是被取代的旧源（与新源冲突时**以新源为准**）。
  - **来源分级**：每条关键说法标来源类型，如 🟢 一手权威 / 🟡 公开可核查 / 🟠 客观常识 / 🔴 占位待补 / ⚫ 作者润色。
  - **口述决策记录**：用户口头敲定的取舍 / 冲突处置，带日期与「旧版 → 新版（以此为准）→ 处理」；AI 改动前先回查，不得擅自推翻。
  - **已删除/降级清单**：曾经"编的"、找不到出处的细节，记下来防止下次又冒出来。
- **`reference.md`** —— 引用关系总表（`[N]` → 文献 → 它支撑的数据点）**＋** 逐数据点核验记录（原「数据核验底稿」并入：每个数据点 `D01..` 的 ✅/⚠️/🚨 结论 + 一手出处）。引用核查与二次核验都围绕它更新。

### 与 ppt-master 配合（含 PPT 的项目）

当项目**同时有 PPT 和 docx/论文**时，PPT 工程由 `ppt-master` skill 负责，但**两侧必须共用同一套资产**，以 `ppt-master` 的工程目录为准对齐：

- `ppt-master` 把图放在 `<project>/images/`（含 `image_manifest.json`、`image_prompts.json`），源文献经 `import-sources --move` 收进 `<project>/sources/`。
- 含 PPT 的项目：本 skill 说的"项目级 `images/` 单一源"**直接复用 `ppt-master` 的 `<project>/images/`**；源文献统一进 `<project>/sources/`（与本 skill 的 `sources/` 是同一个目录）。**不要为 PPT 或 docx 各自再建一个图目录。**
- docx/论文内嵌图、PPT 的 SVG 引用都指向同一个 `<project>/images/`；任一方更新图后，另一方用刷新脚本同步。
- 这样 PPT、docx、论文的引用编号、文献条目、配图资产都落在同一份数据基准上，多 deliverable 自然一致。

### 刷新脚本骨架

docx 内嵌图刷新（基于 `references/doc-report.md` §2.3《替换内嵌图》）：

```python
# scripts/sync_docx_images.py
from docx import Document
from pathlib import Path

def sync_docx(docx_path: str, image_mapping: dict[str, str]):
    """
    image_mapping: {内嵌图名 → images/ 下的源路径}
    示例：{'image2.png': 'images/chart-<主题A>-v2.png'}
    """
    doc = Document(docx_path)
    updated = []
    for rel in doc.part.rels.values():
        for inner_name, source_path in image_mapping.items():
            if inner_name in rel.target_ref:
                rel.target_part._blob = Path(source_path).read_bytes()
                updated.append(f'{inner_name} ← {source_path}')
    doc.save(docx_path)
    return updated

# 用法
print(sync_docx('output/汇报.docx', {
    'image2.png': 'images/chart-<主题A>-v2.png',
    'image3.png': 'images/chart-<主题B>-v2.png',
}))
```

pptx 内嵌图同理，用 python-pptx 的 `slide.part.rels` 遍历替换。

### 反例（不要这样做）

- ❌ 在 `ppt/images/`、`docx/images/`、`web/images/` 各放一份图 —— 三处分叉，没人知道谁是最新
- ❌ 直接把图片粘进 docx、不留源文件 —— 改不了也复用不了，未来用 sha256 比对是"事后救火"
- ❌ 文件名 `chart-<主题A>.png`（不带版本号）+ "覆盖"工作流 —— 改完原版回不去
- ❌ 写一个"sha256 比对脚本"作为常规检查 —— 这是事后救火，根因在没有 single source；真正解法是从架构上消除分叉

## 配图与 AI 标注（跨模式）

### 配图与示意图生成（codex-image）

论文/汇报/PPT 需要原创配图（概念示意、流程图、实景插画、数据信息图）时，用 **codex-image** 生成。要点：

#### 用 codex 的意义 = 用它的 image 生成模型

**调 codex-image 的唯一目的，是让它用图像生成模型真正"画"出图**。不要让它退化成"写一段 SVG/matplotlib/PIL 代码拼图"的方案——那种产物美感差、像模板、文字易重叠，失去用 codex 的意义。prompt 里要明确"生成精美的插画/信息图"，需要时直接点明"用图像生成"。

> 例外的现实约束：**纯图像模型画中文/数字几乎必然乱码**。所以"含大量精确中文文字的数据图"是少数可以接受"AI 出精美视觉底 + 程序精确叠字"的折中场景，但也要把"视觉要精致、文字间距充足不重叠"写成硬要求，并逐张核验。其余概念图/实景图/图标，一律走纯图像生成、画面内**不要任何文字**（避免乱码）。

#### 给足背景，让 codex 自己想怎么画好

codex 是 agent，会基于背景自行决定构图。**prompt 要长、要完整**，至少包含：

- **场景与用途**：什么课题、这张图放在论文/PPT 的哪一部分、要论证什么观点。
- **画面内容**：主体、元素、隐喻、各部分关系（越具体越好，但把"怎么构图好看"留给它）。
- **风格与配色**：写实/扁平/科技插画、主色调、是否统一系列风格。
- **尺寸**：横版/竖版、像素（如半屏插图 1280×800，不要整页满版）。
- **硬约束**：无文字/无品牌标识、背景要求（见下）、不要乱码。
- 让它**完成后自检并报告**（尺寸、是否 RGBA、四角 alpha、有无文字）。

#### 并行生成

codex-image **一次调用出一张图**。多张图时**并行发起多个调用**（同一轮多个工具调用），不要串行等。注意：单次让它一口气做多张容易超时；**一张一调、并行多调**最稳。个别调用报 MCP 超时不代表失败，图常已落盘，核验文件即可。

#### 透明背景（PPT 友好）

线条/图标/示意类、主体是图形而非照片的，让 codex 输出**带 alpha 透明通道的 PNG**（无背景色、无背景矩形），方便直接叠在 PPT 任意底色上。codex 有时去底不净（图标自带白圈时），可补一步程序去底：从四角 flood-fill 去连通背景色、保留图标内部白色，再合成到深色底验证干净。

#### 验收

生图后**逐张 view 核验**，别只信 codex 自报：构图是否达意、有没有该透明却白底、数据图数字对不对、有无文字重叠/乱码、中心图形是否规整。不合格就带着**具体缺陷描述**重做（如"左侧 Y 轴标题与刻度数字重叠""中心圆画成了土豆形"），codex 据此修。

> 浏览器/反爬抓真实图、生成图工作流的更多细节见 browser-use skill。

### AI 标注的措辞 gradient

**学校规范常要求"AI 生成/产生的部分必须明确标注"**。标注要**明确但不占题注篇幅**，按"AI 介入程度"分 4 档：

| 介入程度 | 措辞示例 | 何时用 |
|------|------|------|
| **完全 AI 生成** | "（AI 辅助生成）" / "（AI 生成）" | 概念示意图、装饰性插画、无真实数据的信息图 |
| **AI 据真实数据制图** | "数据来源：文献[X]；作者据 X 数据制图（AI 辅助）" | 数据图（柱状/折线/饼） —— 数据真，排版/渲染由 AI 完成 |
| **真实数据 + AI 排版** | "数据来源：文献[X]；作者编辑 + AI 排版" | 复合信息图，作者主导数据选择，AI 做视觉布局 |
| **真实数据 + 手工制图** | "数据来源：文献[X]；作者自绘" | 无 AI 介入的传统制图，**不要标 AI** |

**原则**：

- **不要"一刀切全部标 AI"** —— 会让真实手工图也被怀疑
- **不要"一律不标"** —— 违规
- **题注内一句话表达完**，不要为 AI 说明开新段
- **页脚兜底**：可统一加一句 "部分图表/排版借助 AI 工具生成，已就地标注"
- **默认偏短措辞** —— "（AI 辅助生成）" 比 "（本图由 AI 辅助完成，作者校对）" 更优

## 引用核查与优化（跨模式）

> **开工前必做**：确认要处理的对象 —— 模式 A 论文按 `references/thesis.md` §5《多版本管理》确认版本、模式 B 确认单一 docx、模式 C 确认单一 pptx；存在多个候选时必须使用当前环境可用的提问能力向用户确认。

处理某一章（模式 A）或整段（模式 B/C）时逐段执行。多 AI 并行修订时，**每个实例只负责分配到的范围**，不干预其他范围。

### 核查流程

**① 读原始文献**：在项目内的文献原文目录（如 `mineru/`、`sources/`、`references/` 等）下搜索相关 `.txt`/`.md`/`.pdf`/`.html`，核对论文表述与原文是否一致。

**② 补引用**：对明显引用了某说法或官方文件却未标注的地方，从尾注列表或仓库搜索匹配文献，补充脚注，注明具体页码（专著类）。

**③ 删不当引用**：
- 引用内容与文献不符（过度推断、错误归因）
- 近几节中已引用过相同文献的相同观点
- 引用数量过多影响可读性

**④ 搜索未知文献**：对仓库找不到、也无链接的文献，多轮网络搜索：
- 组合：`作者名 + 论文标题`、`标题关键词 + 年份 + 期刊`
- 搜到后给出与论文相关的论述及原始链接
- 若始终搜不到，标记为"存疑文献"

### CNKI 检索补源经验

为替换低等级或过度集中的脚注时，优先使用 CNKI `kns.cnki.net/kns8s/AdvSearch` 高级检索，不把 `search.cnki.com.cn` 作为主流程。公开搜索入口只适合快速确认题名是否存在；高级检索才能稳定按主题、篇名、作者、来源、CSSCI/北大核心分组筛选。

固定流程：

1. **先高级检索，再精确核验**：用 `主题/篇名/作者/文献来源` 组合检索；找到候选后必须进入 CNKI 单篇页，核对题名、作者、期刊、年卷期、页码、摘要、被引/下载数。
2. **同义词组要展开**：不要只搜用户给的字面词。把一个核心概念扩成一组近义/相关检索词再逐个检索（一个概念往往有"现象描述 / 成因 / 对策 / 评价"等多个角度的同义表述），命中面更广。
3. **筛选比题名更重要**：先看左侧来源类别中的 CSSCI、北大核心，再从结果中挑主题最贴近正文论点的文献；不要为了"题名像"而放弃更权威、更贴合的核心文献。
4. **题录不能靠猜**：作者真实、主题相似、普通网页提到过，都不等于题录真实。报告中只写 CNKI 单篇页可打开、摘要和统计项可见的文献。
5. **PDF 状态要诚实**：CNKI 的 `bar/download/order` 链接通常不能直接复制下载，直接打开可能报"来源应用不正确"；必须在已认证的单篇页点 `PDF下载`。若自动点击未落盘，报告写"需用户认证后手动下载"，不要说已下载。
6. **善用 search.cnki.com.cn 抽取下载链**：`search.cnki.com.cn` 不适合作为主检索，但可作为 PDF 链接补源通道。结果页的小下载图标通常只是打开 `www.cnki.com.cn/Article/...htm`，进入该文章页后再找 `pay.cnki.net/zscsdoc/download?flag=cnkispace&plat=cnkispace&filename=...&dbtype=...&dtype=pdf`。这个链接未认证时会跳到 `mall.cnki.net/loginspace.aspx`，不代表链接无效；用户完成教育/IP/知网空间认证后，应优先尝试这类 `zscsdoc` PDF 链接。
7. **报告按用途排序**：每个替换位置给 1-2 篇高质量候选，写清"用于替代哪条脚注、为什么契合、PDF 是否已落盘"；宁可少给，不硬塞不贴合文献。

### 万方检索与题录元数据

CNKI 高级检索需认证、易被反爬挡住时，**万方（s.wanfangdata.com.cn）是稳妥的题录元数据补源通道**——它的检索结果页直接带「作者-《期刊》+核心标识+年卷期」和摘要，足够拼 GB/T 7714 题录与判断期刊等级。

- **抓取方式**：万方检索页直连返回 **HTTP 200（实测不是 403、无滑块）**，但**题录是 JS 异步渲染**的——`curl`/`urllib` 只能拿到 Vue/webpack 的 SPA 空壳（页面里有 `chunk-vendors`、`window.__…`，但没有「找到N条」和题录文本）。要拿题录，必须用能跑 JS 的浏览器（见 browser-use skill 的 camoufox 方案）`page.goto` 检索页、等渲染后读 `inner_text("body")`。机构网络（如校内 IP）访问会显示「机构已购」、元数据更全。（对比：百度学术直连才是真滑块验证页，万方不是。）
- **解析按行**：结果是「`N.篇名` / `[期刊论文]作者-《期刊》核心标识 年X期` / `摘要：…`」三行一组，按行正则切，不要指望 class 名（万方 DOM class 会混淆）。
- **筛核心**：结果文本里直接含 `北大核心 / CSCD / CSSCI / EI / CSTPCD` 标识，据此过滤出高等级文献；页面左侧也有「只看核心」「被引/下载」分面。
- **题录仍要核**：万方给的年卷期偶有缺页码；正式定稿前用户应在知网/万方单篇页补全卷(期):页码。新刊（如当年最新一期）卷期可能未最终编定，按「年(期)」最简式著录、不要编造页码。

### 如何确定期刊等级

判断一条文献够不够"高水平"，看它命中哪些**收录/分级标识**（同一刊常多个并列）：

- **CSSCI**（南大核心，人文社科最权威）、**北大核心**（《中文核心期刊要目总览》）、**CSCD**（中国科学引文数据库，理工医）、**EI**（工程索引）、**SCI**（理工医国际）、**AMI**（社科院核心）、**CSTPCD**（科技核心/统计源，等级低于前述）。
- 学位论文报告里，**院士/领域权威作者 + CSSCI/北大核心/CSCD** 的组合最稳；只挂 CSTPCD 的算一般期刊，能换则换。
- 万方/知网结果页的核心标识可直接读；拿不准时进单篇页看「期刊荣誉/收录」。**不要凭刊名猜等级**。

### 数据与事实必须有一手来源

论文里凡是**具体数字、政策表述、官方结论**，脚注/尾注要追到**发布主体的一手出处**，不要停在二手汇编（行业白皮书、新闻转述、百科）：

- **政策/规划/文件** → 政府门户原文（如 `www.gov.cn`、部委官网），著录 `[A/OL]` 或 `[R/OL]` + 发文字号 + 发布日期 + URL + 引用日期。
- **统计数字** → 发布机构原文（统计局、行业主管部委、权威发布会实录），不要引"某文章说"。
- **市场预测/第三方数据** → 报告原始出处（机构 + 报告名 + 年份），原链接失效就按 `[R]` 著录不挂死链。
- **领导人讲话** → 收入《求是》《选编》等正式文本，按报刊/期刊著录。
- **软数据**（来源不可考、仅广泛流传的比例/排名）→ 措辞弱化为"据行业统计"，**不编号、不当硬引用**，避免给不实数字背书。

一手来源查不到时如实标"存疑/待核"，宁缺毋编。

### 上下文逻辑核查

结合上下文逐段检查：
- 是否把原文的"转变/演进"推断成了"失效/否定"
- 是否无原文依据地给原作者添加了论断
- 是否存在逻辑性或结构性表述错误

**所有修订完成后，必须独立做一遍"逐数据点回原文"核验** —— 详细流程见 `references/doc-report.md` §3《完工前二次核验流程》。

### 顺序编码制（文中上标 + 文末交叉引用）

skill 默认走"脚注内联"形态；但很多论文/汇报要的是 **GB/T 7714-2015 顺序编码制**——正文引用处标**上标方括号序号** `[N]`，文末「参考文献」**按正文出现先后顺序**编号列出（GB/T §10.1）。两种制度要点：

- **按出现顺序编号**：序号是"第几个被引到"，不是按作者/年份排。同一处引多篇用逗号 `[1,4]`，连续区间用短横 `[3-5]`（GB/T §10.1.2）。同一文献多处复用同一个号。
- **一处声明、全篇贯穿**：写作时给每条来源一个稳定标记（如 `⟦4⟧` 占位），落版时统一替换成上标 `[4]`，避免中途插引用后手工重排错号。**详细工作流见下《占位符工作流》**。
- **文末条目按类型标识著录**：期刊 `[J]`、专著 `[M]`、报告 `[R]`、电子 `[EB/OL]`、政策档案 `[A/OL]`、报纸 `[N]`、学位论文 `[D]`、标准 `[S]`（GB/T 附录 A 有各类型范例，见 `assets/GBT7714-2015.md`）。

### 生成 docx 时怎么做到"可点击跳转 + 自动编号列表"

（以下机制实测可用，pandoc 反解可验证）

1. **文末参考文献 = Word 自动编号列表**（不是手打 `[1] [2]`）。在 `word/numbering.xml` 定义一个 `<w:abstractNum>`（含 `<w:lvl>`：`numFmt=decimal`、`lvlText="[%1]"` 即显示成 `[1]`、`<w:ind left/hanging>` 做悬挂缩进）+ 一个 `<w:num>` 引用它；每条参考文献段落 `<w:pPr><w:numPr><w:ilvl val="0"/><w:numId val="…"/></w:numPr>`。Word 自动渲染 `[1][2][3]`、自动悬挂缩进，**中途增删条目自动重排号**。
   - ⚠️ schema 坑：`numbering.xml` 里**所有 `<w:abstractNum>` 必须排在所有 `<w:num>` 之前**（CT_Numbering 的 sequence 约束）；用 python-docx 操作时把 abstractNum `insert` 到最前、num `append` 到最后。
2. **文末每条加书签**：段落里包 `<w:bookmarkStart w:id="N" w:name="ref_N"/> … <w:bookmarkEnd w:id="N"/>`（书签名如 `ref_1`）。
3. **正文上标 = 指向书签的内部超链接**：用 `<w:hyperlink w:anchor="ref_N">` 包一个上标 run（`<w:rPr>` 里加 `<w:vertAlign w:val="superscript"/>` + `<w:rStyle w:val="Hyperlink"/>`），run 文本写 `[N]`。`anchor` 属性即内部跳转，**不需要 r:id**（那是外链才要）。点正文 `[N]` 就跳到文末 `ref_N`。

三者对应靠"出现顺序"：正文按先后写 `[1][2]…`、文末按同序排，自动编号与上标数字自然一致。落版前用脚本核对 **正文上标个数 == 文末条目数**。

**完全自动（连正文序号也自动重排）** 需用 Word 原生「插入→交叉引用→编号项」或 SEQ/REF 域 + EndNote/Zotero，python-docx 难干净生成、且打开要按 F9 更新域；学位论文一般用上面的"自动编号列表 + 书签超链接"半自动方案即可（文末全自动、正文用占位符统一替换），最稳。

### 删除/插入文献后的检查清单（紧急修复用）

**没用下《占位符工作流》的项目**遇到"删/插一条文献"时，必须做的 4 件事：

1. **删尾注条目** —— 该 `[N]` 整行删除
2. **正文上标处理** —— 按情况：
   - 删整句（论据不要了）
   - 去脚注、保句子（论据保留但不引）
   - 换源（找新文献顶替）
3. **后续条目重编号** —— `[N+1] → [N]`、`[N+2] → [N+1]`...，正文上标和尾注同步改
4. **散落引用别漏** —— `[3, 18]` 这种多文献引用、`[3-5]` 这种区间，扫 `\[\d+[\d,\s\-]*\]` 模式

工程脆弱，**更好的根本解法是下《占位符工作流》**。

### 占位符工作流（顺序编码制的根本解法）

> 顺序编码制下，**删/插任何一条文献都会改变其余所有引用号** —— 直接在正文写 `[N]` 是工程脆弱性。本节给出根本解法：写作时用人类可读 key 占位，落版时一键替换成正确的 `[N]`。

#### 占位符约定（人类可读 key）

```
正文写：⟦han2025⟧、⟦liu2024⟧、⟦han2025, liu2024⟧
落版替换：⟦han2025⟧ → [N]（按 key 在文中首次出现的顺序自动编号）
```

key 命名建议：`<第一作者拼音><年份>`，与 BibTeX / Zotero 兼容。

#### 落版脚本骨架

```python
# scripts/finalize_citations.py
import re
from pathlib import Path

src = Path('manuscript.md').read_text(encoding='utf-8')

# 1. 按出现顺序提取所有占位符 key（保持首次出现的顺序）
keys_in_order = []
seen = set()
for m in re.finditer(r'⟦([a-zA-Z0-9_,\s]+)⟧', src):
    for k in re.split(r'\s*,\s*', m.group(1).strip()):
        if k and k not in seen:
            keys_in_order.append(k)
            seen.add(k)

# 2. key → [N] 映射
key_to_num = {k: i+1 for i, k in enumerate(keys_in_order)}

# 3. 全文替换（处理单 key 和多 key 两种）
def repl(m):
    keys = re.split(r'\s*,\s*', m.group(1).strip())
    nums = sorted({key_to_num[k] for k in keys})
    return '[' + ','.join(str(n) for n in nums) + ']'
out = re.sub(r'⟦([a-zA-Z0-9_,\s]+)⟧', repl, src)

Path('manuscript-final.md').write_text(out, encoding='utf-8')

# 4. 一致性检查
print('编号映射：')
for k, n in key_to_num.items():
    print(f'  [{n}] ← ⟦{k}⟧')
```

#### 落版前一致性检查

- 每个 `⟦key⟧` 在文中出现 ≥1 次（≥1 是正常复用；0 次说明定义了没用 → 删尾注条目）
- 每个 `[N]` 在尾注列表里有对应条目（数量一致 + 编号连续无跳号）
- 多文献引用 `[3, 18]` 由占位符 `⟦han2025, liu2024⟧` 自然生成

#### 优点

- 增删任意文献时只动 key 层面，编号全自动重排
- 文中可读性高（`⟦han2025⟧` 一眼知道是谁的 paper）
- 不依赖 Word 域 / 自动编号列表的复杂机制
- 可与 BibTeX / Zotero key 兼容（直接 `\cite{han2025}` 也可）

#### 与 Word 自动编号列表的关系

- **占位符工作流** 适用于：md 写作 + 最后转 docx（模式 A 论文常用、docx 短文用占位符 md 中间格式也行）
- **Word 自动编号列表**（上一节）适用于：直接在 docx 里写作（模式 B 单章交稿）
- 两者可叠加：md 写完用占位符脚本生成 `[N]`，再 pandoc 转 docx 时把尾注列表挂上自动编号列表 + 书签超链接。

- **文件**：F0–F4（约定见 `references/thesis.md` §4.1）
- **当前行**：当前 working tree 中的行号区间，便于用户跳转
- **原文片段**：报告标记区内的关键短句（≤20 字），便于用户在不打开文件时也能快速对位
- **改前→改后(简+依据)**：两部分。① ≤30 字摘要，区分"删除 / 句式重写 / 换词 / 补脚注"；② 紧跟一行写改动原因或依据，例如「依据：XX 文献原文用'演变'而非'失效'」「依据：避免与 F2 第 88 行重复表述」「依据：报告段 7 共 133 字 U 直接命中」。原因不写或只写"降重"视为信息不足。
- **命中报告段**：如「第2章上·段7」，对应 `full.md` 里的 `### 段 N`
- **相似字数**：报告中"此处有 X 字相似"的 X，纯数字
- **U/Q**：报告标记类型（定义见《名词表》）
- **评价**：一句话说明改得对症 / 不到位 / 应回退 / 待补

排序：A → B → C → N → 缺漏。

输出后同样**只给意见、不动文件**；用户确认后由新上下文中的 AI 逐条应用（见 `references/thesis.md` §4.2 工作流）。

### 重复说法清单（`references/thesis.md` §3.2 用）

```
| 组号 | 类型 | 关键短语/文献 | 出现位置 | 上下文摘要 | 初判 |
| --- | --- | --- | --- | --- | --- |
```

字段说明：
- **组号**：G1, G2, …，便于后续逐组在新上下文中处理
- **类型**：关键词反复 / 同义反复段 / 跨章节同源引用
- **关键短语/文献**：被重复使用的短语，或被反复引用的文献简称
- **出现位置**：F1:120-122; F3:45-46; F4:88-90 这种格式，列全
- **上下文摘要**：每处 ≤20 字，方便用户一眼区分各处侧重
- **初判**：冗余可合并 / 应差异化保留 / 仅删某一处 / 待用户判断

排序：按"明显冗余 → 应差异化 → 待判断"递减。

每组用户确认后，单独开新上下文处理这一组（流程见 `references/thesis.md` §3.2）。
