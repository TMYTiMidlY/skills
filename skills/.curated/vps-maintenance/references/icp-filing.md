# ICP 备案与端口策略

> 经验性总结，不是法律意见。内容融合了自有的大陆 ECS / 家宽实测（如 2026-04 阿里云 ECS 的 SNI 封锁实测）和 2026-05 与同好交流补充的备案豁免边界，并对照阿里云官方备案 FAQ 厘清「备案 vs 接入备案」；具体合规边界以监管和运营商最新口径为准，长期对外服务请以备案/合规为目标。

国内自托管经常被同一个问题卡住：“这个东西要不要备案？” 这份 reference 把常见组合归类，便于快速判断；以及由这些约束衍生出的部署模式。

## 核心机制

国内云 IDC（含阿里云大陆 ECS 等）的链路上有**针对 TLS ClientHello SNI 的旁路检测**：握手发出去之前，链路设备先读明文 SNI，匹配到未备案域名就直接 TCP RST。**与端口号无关**——80/443、8443、18082、28082 实测形态完全一致（2026-06 本机+境外双向 openssl 测试：未备案 SNI → `write:errno=104` + 0 字节应答；同链路换成已备案 SNI（baidu / aliyun）→ 穿过 RST 到达 Caddy，正常拿到 7 字节 TLS alert）。

**所以“未备案域名 + 国内云 ECS”的任何 TLS 端口都跑不通**——这是技术问题，不只是合规问题。

**不在这条路径上的（不受 ICP 影响）**：

- DNS 解析（不同协议层，由域名注册商/DNS 服务商负责，与 ICP 无关）
- SSH（含 `git@host` 的 git-over-SSH，本质就是 `ssh git@host git-upload-pack ...`）
- 其他非 TLS 协议（SMTP、WireGuard、自定义 TCP 服务等）——没有 SNI 字段，没法被基于 SNI 的旁路检测匹配
- 不带 SNI 的纯 IP TLS 访问（浏览器直连 IP 时默认不发 SNI，RFC 6066 也不允许 IP 作 SNI）
- 海外 VPS（链路上没有这条旁路检测设备）
- **家宽公网 IP 上的非标端口 HTTPS**——能用**不是因为非标端口本身豁免**，而是因为家宽链路上没部署这条旁路检测；同样的请求换到国内 ECS 上一样被拦

### 备案绑在域名上，不绑服务器账号

前面说链路设备读 SNI 里的域名、未备案就 RST。反过来推一步：它**只**看 SNI 里的域名、按域名比对备案状态，根本看不到也不关心这台机器是谁、用哪个云账号买的。所以“过不过 RST”只有一个维度——**SNI 里的域名是否已备案**：

- 备案号是「域名（网站）+ 主体」维度的属性，由管局签发、挂在某个接入商名下，跟“机器是谁买的”无关。阿里云官方备案 FAQ 里就直接列着“备案可以使用他人的服务器吗”“我购买的服务器可以让别人备案吗”“在阿里云备案必须购买服务器吗”这些条目，可见备案号和服务器购买账号本来就是解耦的。
- 推论：**只要 SNI 是已备案域名，把它解析到任意账号（哪怕别人账号、别的云厂商）买的国内云服务器，都能穿过链路 RST**。上文实测正是佐证——拿 `baidu` / `aliyun` 这种别人备案的域名当 SNI 打到自己阿里云 ECS 上，照样穿过 RST 到了 Caddy（只是没匹配站点、回了 alert）。真用时把“自己的已备案域名”指过去即可。

> ⚠️ **接入备案是另一层：行政手续，不是访问拦截**。备案号挂在某个接入商名下，换一家厂商用得补一次“接入备案”（接入商变更 → 重新申请接入，流程类似新备案——阿里云开发者社区就有人问“有备案号的域名想用阿里云一定要接入吗”，答复是“必须重新接入使用”“接入商有变更所以需要申请接入”）。但它**不卡访问通不通**：链路只按域名备案号放行，已备案域名解析到任意厂商的国内服务器日常一般照常能用（与“接了别人备案的域名也没出过问题”的经验一致）。真正的约束是厂商/管局的**接入核查**——长期把别处备案的域名挂在某厂商服务器却不在它家接入，可能被核查、要求补接入，甚至被取消接入、备案被管局注销。所以：**只图能跑通常无感，要干净长期合规就补一次接入备案**。

## 速查矩阵

| 场景 | 是否需要备案 | 关键约束 |
|---|---|---|
| DNS 解析任何域名 | 不需要 | DNS 由域名注册商/DNS 服务商负责，与 ICP 无关 |
| SSH / 其他非 TLS 协议（含 `git@host`、SMTP、WireGuard 等） | 不需要 | 没有 SNI 字段，不在旁路检测路径上 |
| 海外 VPS + 自有域名 | 不需要 | 监管管不到海外资源 |
| 国内云 ECS + 自有**未备案**域名 + **任何 TLS 端口**（80/443/非标端口都一样） | **需要**，否则**技术上跑不通**（中间设备读 SNI 直接 RST） | 详见下文“SNI 封锁实测”；端口号无关 |
| 国内云 ECS + **已备案**域名（备案挂谁名下/哪个账号都行）+ 任何 TLS 端口 | 不需要重复备案，链路 RST 能过；**不要求与服务器同账号** | 链路只认域名备案号即放行；换到别家厂商用属行政手续（接入备案 / 接入核查），不实时卡访问 |
| 国内云 ECS + 纯 IP（不带域名） | 不需要 | 浏览器纯 IP 访问不发 SNI，没有域名可匹配；用 `tls internal` + 客户端装 root CA |
| **家宽公网 IP** + 自有域名 + **非标端口** | 不需要 | 家宽链路上不部署这条 SNI 旁路检测，所以非标端口能正常签 Let's Encrypt 也能正常访问；运营商通常封 80/443，所以本来也只能走非标端口 |
| 家宽公网 IP + 80/443 | —— | 大多数家宽运营商**直接封 80/443 入站**，技术上跑不起来 |

> 矩阵里“不需要”的几条都是基于自建圈普遍口径与个人实践（V2EX 等），不代表监管规则。规模一大就建议老老实实备案。

## SNI 封锁实测（国内云 ECS）

2026-04 + 2026-06 在阿里云大陆 ECS 上的实测观察：

- HTTPS 封锁看的是 TLS ClientHello 里的 SNI（明文），即目标域名。只要请求里带了未备案的域名，就会被拦——**与端口号无关**。
- 最初用 `https://<域名>:8443` 访问是正常的；经过多次测试之后，家宽访问同一地址返回了阿里云“要求备案”的提示页，手机流量访问仍然正常。是不是反复测试导致命中的不确定，反正结果是家宽这边被拦了。
- 2026-06 用 `openssl s_client` 直接复现并定位拦截位置：未备案 SNI 在 `:18082` / `:28082` 等任意非标端口都返回 `write:errno=104` + 0 字节应答（链路 TCP RST，没到 Caddy）；同样的连接换成已备案 SNI（baidu / aliyun）就能穿过 RST，由 Caddy 正常发回 7 字节 TLS alert（没匹配站点）。境内境外双向形态一致，证明是**链路上中间设备的旁路 SNI 检测**，不是 Caddy 本身拒。

### 当前可用方案：IP 直连 + Caddy internal CA

结论：只要请求里不带域名就没事。浏览器用纯 IP 访问时默认不发 SNI，也没有域名可匹配，实测稳定可用，是目前在用的方案。

一个服务一个端口，`https://<主 IP>` 走默认的 443 也能正常访问；其他服务用各自的端口。TLS 统一 `tls internal` 自签，客户端装 Caddy local root CA，见 [caddy.md](caddy.md) 的“导入 Caddy local root CA（仅 `tls internal` 场景）”小节。

> 严格讲这是违规操作，V2EX 上有反馈过“一个星期后接到电话”的案例，共识是小规模用通常被容忍。长期要合规还是得备案或走海外前端反代。

## 家宽公网 IP + 非标端口 自用 HTTPS

只要满足以下三点，自用 HTTPS 服务在国内家宽公网 IP 上是常规做法：

1. **走非标端口**（如 `:10003`、`:8443`），绕开运营商对 80/443 的入站封锁。
2. **域名解析到自己的 IP**（A / AAAA 记录），不经过国内云厂商也不在国内 CDN 上。
3. **自用规模**，不对公众宣传/推广。

证书直接走 Let's Encrypt 就行——家宽链路上没有那条 SNI 旁路检测，非标端口的 HTTPS 既不影响签发也不影响访问。范例形式：

```
https://ddns.shenjack.top:10003/dashboard
```

⚠️ **不要把这条经验外推到国内 ECS**——同样“未备案域名 + 非标端口”，放到阿里云大陆 ECS 上就会被中间设备 RST（见上文“SNI 封锁实测”）。**链路不同，机制不同**：家宽能用是因为家宽链路上不部署这条旁路检测，**不是因为“非标端口豁免 SNI 拦截”**。

> 补充：“**有域名 + 非标端口正常签 Let's Encrypt**” 跟 [caddy.md](caddy.md) 的“IP 模式（无域名 / 未备案）”是互补关系：caddy.md 教的是“没有域名时用 `tls internal` + 客户端装 Caddy local root CA 凑合”；这里讲的是“有域名时用真实 CA 签发的非标端口 HTTPS”——后者**只对家宽/海外 VPS 适用**，对国内 ECS 仍要走 IP 模式或备案。两条路按 链路是否带 SNI 旁路检测 + 是否能接受导入 root CA 来选。

## 由备案约束衍生的部署模式

家宽公网 IP 在国内**几乎必然**遇到两个限制：

- 80/443 入站被运营商封；
- NAT/CGNAT 让某些设备根本拿不到真公网 IP。

为此常见的妥协做法是**额外买一台小的国内 VPS 做反代/组网中转**：

- 国内 VPS 拿一个公网 IP，跑 EasyTier / frp / WireGuard，把家里设备拉进虚拟网络。
- 国内 VPS 上**不要用未备案域名对外暴露 HTTPS**——非标端口照样会被 SNI 旁路检测 RST（见“核心机制”）。两条可行路径：
  - **IP 直连模式**：用 `tls internal` + 客户端装 Caddy local root CA，请求里不带域名 → 不触发 SNI 拦截。
  - **已备案域名**：拿任意一个已备案域名（备案挂谁名下都行，已在别处备案的走一遍接入备案即可）解析到这台国内 VPS，正常做 HTTPS——备案绑的是域名、不绑服务器账号（见“核心机制”）。
- 国内 VPS 的好处是延迟低、对国内访问友好；缺点是受备案/审计约束更强，所以**只跑组网中转和反代落地，不做面向公众的服务**。

EasyTier 的具体配置见 [easytier.md](easytier.md)；Caddy 反代见 [caddy.md](caddy.md)。
