# Plan

## 何时使用

用户表达以下意图时触发：

- "写一个方案让别人来实现"
- "写成一份给另一个 AI / 智能体执行的文档"
- "做一份正式实施规划，后面按这个来做"

以下情况**不触发**本 skill：

- 用户只想要一个简短计划、行动项或步骤拆分
- 用户只是说"帮我理一下思路"、"简单规划一下"
- 用户要的是你当前回合自己继续实现前的内部计划

如果只是任务复杂，但用户**没有**明确要求要一份正式实施文档：

- 不自动触发本 skill
- 先询问用户是否希望先出一份**正式实施规划文档**，再决定是否进入本 skill

## 产出物要求

输出一份**面向 AI 智能体实施者**的正式实施文档。实施者没有本次会话上下文，文档必须自包含。

如果用户只想要简短规划，不应使用本 skill。

## 核心原则

### 分步与可验证

- 每个步骤独立可验证——给出具体的验证命令或检查方法，执行后能明确判断"通过/不通过"。
- 步骤之间标明依赖关系和执行顺序。
- 不要把多个独立关注点塞进同一步骤。

### 讲设计，不只讲代码

- 每个设计决策解释**为什么这样做**，不只说做什么。
- 标注备选方案和取舍理由。
- 列出容易踩的坑和需要注意的边界条件。
- 如果有对现有代码的依赖或约束，说明清楚（包括调用了哪些内部方法、为什么可以这样调用）。

### 错误场景覆盖

- 对用户可感知的功能，列举典型错误场景和预期行为。
- 错误信息要对人类有用，不是裸 traceback。

### 灵活性设计

- 通过环境变量或配置暴露切换点，而非硬编码。
- 考虑不同部署场景（如：组件在同一台机器 vs 分布在不同机器）。
- 代码中不绑定具体的第三方服务/产品名称，用抽象术语。

### 私有环境信息分层

- 默认不要把维护者私有环境的具体细节写进主方案，例如域名、IP、端口、认证头、内网拓扑、隧道产品名等。
- 只有当这些信息**直接影响设计决策**时，才在规划中提及，并尽量抽象为"私有部署环境"、"现有认证网关"、"当前运行约束"等表述。
- 如果确实需要保留具体私有信息，先询问用户是否要写入，并优先放在"部署附录"、"维护者注记"或独立备注中，不要混进通用实施文档主体。

### 可复用性

- 设计要让其他有类似需求的人 fork 后能直接用。
- 明确哪些是项目特定的、哪些是通用的。

### 额外的思考

- 不要默认加入"额外的思考"。
- 如果你认为补一节未来演进方向有价值，先询问用户是否希望加入。
- 用户同意后，再设计一两个实现起来可能较重、但绑定在项目核心流程中、不适合拆出去的方向。
- 这些内容标明"不在本次实施范围"，但为未来演进提供锚点。

## 文档结构模板

```
# {项目名} {功能名}：实施规划

## 需求与背景
（问题是什么、约束是什么）

## 现有代码/基础设施分析
（读代码后的发现、现有机制的问题）

## 整体架构设计
（图示 + 部署模式 + 关键边界）

## 环境变量 / 配置设计
（完整变量清单 + 每个变量的设计考量）

## 实施步骤
（每步：改什么文件、大致代码思路、设计注意、验证命令）

## 安全边界
（威胁 → 缓解措施 → 实现位置）

## 额外的思考
（不在本次范围，但与项目核心流程绑定的方向）

## 关键设计注意事项
（踩坑点、易错点、代码风格要求、不应修改的文件清单）
```

不必严格遵循此模板——按实际需求增删章节。但这是**正式实施文档** skill，产出物应保证"分步实施 + 每步验证"和"设计考量"完整可用。

## 工作流程

1. **充分调研**：读代码、读配置，并在当前环境允许时核实真实运行环境与部署约束。不要凭假设规划。
2. **与用户对齐**：不确定的设计决策先问用户，不要自作主张。若任务只是复杂但用户没明确要求正式实施文档，先确认是否需要这种文档。
3. **确认附加内容**：若你认为应加入"额外的思考"或私有部署附录，先询问用户。
4. **写规划**：按上述原则输出正式实施文档。
5. **不写实现代码**：规划阶段只产出文档。代码片段仅用于说明设计意图，不要写完整实现。
