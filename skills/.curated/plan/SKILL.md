# Plan

## 何时使用

用户表达以下意图时触发：

- "先规划/设计，不要写代码"
- "写一个方案让别人来实现"
- "帮我拆分成步骤"
- 或者任务复杂到你判断应该先规划再动手

## 产出物要求

输出一份**面向 AI 智能体实施者**的文档。实施者没有本次会话上下文，文档必须自包含。

## 核心原则

### 分步与可验证

- 每个步骤独立可验证——给出具体的验证命令或检查方法，执行后能明确判断"通过/不通过"。
- 步骤之间标明依赖关系和执行顺序。
- 不要把多个独立关注点塞进同一步骤。

### 讲设计，不只讲代码

- 每个设计决策解释**为什么这样做**，不只说做什么。
- 标注备选方案和取舍理由。
- 列出容易踩的坑和需要注意的边界条件。
- 如果有对现有代码的依赖或约束，说明清楚（包括调用了哪些内部方法、为什么可以这样调用）。

### 错误场景覆盖

- 对用户可感知的功能，列举典型错误场景和预期行为。
- 错误信息要对人类有用，不是裸 traceback。

### 灵活性设计

- 通过环境变量或配置暴露切换点，而非硬编码。
- 考虑不同部署场景（如：组件在同一台机器 vs 分布在不同机器）。
- 代码中不绑定具体的第三方服务/产品名称，用抽象术语。

### 可复用性

- 设计要让其他有类似需求的人 fork 后能直接用。
- 明确哪些是项目特定的、哪些是通用的。

### 额外的思考

- 在文档中设一节"额外的思考"，为项目设计一两个实现起来可能较重的功能模块，此模块应该是绑定在项目核心流程中、不适合拆出去的设计方向。
- 这些内容标明"不在本次实施范围"，但为未来演进提供锚点。

## 文档结构模板

```
# {项目名} {功能名}：实施规划

## 需求与背景
（问题是什么、约束是什么）

## 现有代码/基础设施分析
（读代码后的发现、现有机制的问题）

## 整体架构设计
（图示 + 部署模式 + 关键边界）

## 环境变量 / 配置设计
（完整变量清单 + 每个变量的设计考量）

## 实施步骤
（每步：改什么文件、大致代码思路、设计注意、验证命令）

## 安全边界
（威胁 → 缓解措施 → 实现位置）

## 额外的思考
（不在本次范围，但与项目核心流程绑定的方向）

## 关键设计注意事项
（踩坑点、易错点、代码风格要求、不应修改的文件清单）
```

不必严格遵循此模板——按实际需求增删章节。但"分步实施 + 每步验证"和"设计考量"是必须有的。

## 工作流程

1. **充分调研**：读代码、读配置、必要时 SSH 到远端查看实际环境。不要凭假设规划。
2. **与用户对齐**：不确定的设计决策先问用户，不要自作主张。
3. **写文档**：按上述原则写实施规划。
4. **不写实现代码**：规划阶段只产出文档。代码片段仅用于说明设计意图，不要写完整实现。
