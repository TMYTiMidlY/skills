## 1. 文档格式转换

使用 `scripts/doc_to_md.py` 将 `.doc`/`.docx` 转换为分章 Markdown：

```bash
uv run scripts/doc_to_md.py <论文文件.doc>
```

**输出结构**（存入同名目录，原文一同保存）：
```
<论文名>/
├── 00_前言与参考文献.md      # 封面/摘要/目录 + 参考文献列表
├── 01_第1章_<章节名>.md
├── 02_第2章_<章节名>.md
├── ...
└── <原始文件名>.doc
```

**处理逻辑**：
- .doc 文件先用 LibreOffice 转为 .docx，再用 pandoc 转 md
- `[^N]` 脚注引用展开为正文内 `（脚注：…）` 格式
- 尾注参考文献保留为编号列表
- 去除 pandoc 产生的多余 `\[` `\]` 转义

## 2. 脚注与尾注规范

### 格式

**脚注**（正文中具体引用）：直接内联在引用位置后方：
```
作者指出某观点（脚注：作者.题目[J].期刊,年份,(期):页码.）
```

**尾注参考文献**（全文隐性参考）：`00_前言与参考文献.md` 末尾的编号列表，按 GB/T 7714-2015 格式。

### 脚注 vs 尾注的关系

- **脚注**：明确引用了某段话、首次引入某名词或观点时使用
- **尾注**：全文隐性参考；期刊类需标注页码，其他通常不需要
- 二者**无包含关系**：一篇文献可同时出现在脚注和尾注
- 多次引用同一文献的观点，可写进尾注，表示更多地方参考了它但未逐一标注
- **更新尾注时：凡出现过脚注的文献，都应包含在尾注中**

### 脚注位置原则

- 附近一两节内若已引用过同一文献的同一观点，**不重复引用**
- 明确引用官方文件或政策、首次引入关键概念时必须有脚注
- 专著类脚注须包含具体引用页码

## 3. 全文结构通读与内部去重

> **开工前必做**：按 §7 确认要处理的论文版本，存在多个候选时必须使用当前环境可用的提问能力向用户确认。

在做任何具体引用核查或降重之前，先建立"全文一张图"的视野。论文是一个整体，孤立地处理某一段很容易忽略章节之间的承接、重复与冗余。

### 3.1 结构校验：通读目录与脉络梳理

目的：让 AI 形成对论文骨架的整体认知，后续每一处修改都能基于"这段在全文中承担什么"判断改/不改。**本环节不直接动文件**，只输出结构判断给用户审。

固定动作：

1. **读 F0 的目录与摘要**，掌握章节划分与全文论点。
2. **抽样读各章首尾**：每章读首段（导入/章引）+ 每节首句 + 章末小结，绘制章节脉络（哪一章为哪一章铺垫、哪两章是并列论证、哪一章是收束）。
3. **检查结构问题**：
   - 章节标题与实际内容是否匹配
   - 是否存在论证跳跃或前后呼应缺失
   - 是否有应在某章出现却被错放到另一章的内容
4. **形成结构脉络说明**（一段或一张简表）反馈给用户，等用户确认结构判断后再进入后续工作。

### 3.2 内部重复说法与同源引用检查

目的：避免完全一样的内容在不同章节反复出现，避免同一文献在远距离章节被反复引用却没有差异化论证。这两类问题查重报告**查不出来**（不属于外部相似），但严重影响论文质量与可读性。

检查面：

- **反复出现的关键词/表述**：用 Grep 在全文范围搜索高频核心短语（如"育人职能""三全育人""双线晋升"等），列出每一次出现的章节、行号与上下文。
- **跨章节同源引用**：扫脚注，找同一文献在不同章节、且物理距离较远的多次引用，判断是否在论述同一观点。
- **同义反复段**：一段话在两章里换了几个词又说一遍，没有新增侧重。

工作流（同样**不直接动手改**）：

1. AI 输出"重复点清单"——每一组列出：关键词/短语/文献、各处出现位置（文件:行号）、上下文摘要、初步判断（是否冗余）。
2. 用户确认哪些需要处理。
3. **每一组单独在新上下文中处理**：让新的 AI 实例只看这一组的所有出现位置和相关引文，由用户决定每处的侧重点（如：在第 2 章谈"政策演变"、在第 3 章谈"驱动机制"、在第 4 章谈"实践问题"），AI 据此差异化改写或合并删除。
4. 处理完一组再开下一组，避免一次性大改造成失控。

**强提醒：去重不是机械的"换词游戏"，是概念辨析。** 处理每一组前 AI 必须：

- **多读引文**：把这一组涉及的所有原始文献完整读一遍（不只是查重报告摘出的几句），理解作者的论述脉络与术语用法；只看一句话很容易把"演变""转型""重塑"这种近义但侧重不同的词用错。
- **多读上下文**：每个出现位置至少向上向下读 30 行，看清这一段在所在章节承担的论证角色，避免按字面意思改完后破坏论证链。
- **多轮对话明晰概念**：遇到"这两处到底是同一个概念还是相似但不同"的判断时，**主动使用当前环境可用的提问能力与用户对话 2–3 轮**，把每处的侧重点、对应理论来源、与上下文的关系问清楚，再下手；宁可多聊几轮，也不要靠 AI 自己猜。
- **暂停的判断**：如果对某概念的边界仍不确定，宁可在评估表里标"待用户补充说明"暂停这一处，也不要勉强改写。

输出格式见 §8「重复说法清单」。

## 4. 引用核查与优化

> **开工前必做**：按 §7 确认要处理的论文版本，存在多个候选时必须使用当前环境可用的提问能力向用户确认。

处理某一章时逐段执行。多 AI 并行修订时，**每个实例只负责分配到的章节**，不干预其他章节。

### 核查流程

**① 读原始文献**：在项目内的文献原文目录（如 `nlm/`、`references/`、`mineru_output/` 等）下搜索相关 `.txt`/`.md`，核对论文表述与原文是否一致。

**② 补引用**：对明显引用了某说法或官方文件却未标注的地方，从尾注列表或仓库搜索匹配文献，补充脚注，注明具体页码（专著类）。

**③ 删不当引用**：
- 引用内容与文献不符（过度推断、错误归因）
- 近几节中已引用过相同文献的相同观点
- 引用数量过多影响可读性

**④ 搜索未知文献**：对仓库找不到、也无链接的文献，多轮网络搜索：
- 组合：`作者名 + 论文标题`、`标题关键词 + 年份 + 期刊`
- 搜到后给出与论文相关的论述及原始链接
- 若始终搜不到，标记为"存疑文献"

### 逻辑审查

结合上下文逐段检查：
- 是否把原文的"转变/演进"推断成了"失效/否定"
- 是否无原文依据地给原作者添加了论断
- 是否存在逻辑性或结构性表述错误

## 5. 降重与查重应对

> **开工前必做**：按 §7 确认要处理的论文版本，存在多个候选时必须使用当前环境可用的提问能力向用户确认。

围绕查重报告（推荐用 `scripts/查重_html_to_md.py` 生成的全文对照报告单 `full.md`）开展降重与补引。两条主线：**降低重复率**（改写 U/Q 标记句的句式）+ **增加正常引用**（对未标注的整段照搬补脚注）。

**色标 → 字母约定**（全文统一）：
- **U** = Unquoted，对应报告中的【红】（未引证、计入"去除引用复制比"）
- **Q** = Quote，对应报告中的【绿】（已标脚注，仅计入总复制比、不计入"去除引用复制比"）

### 文件简称约定（写死）

降重相关的所有沟通、表格、行号引用一律用以下简称，避免反复粘贴长文件名：

| 简称 | 文件 |
|------|------|
| F0 | `00_前言与参考文献.md` |
| F1 | `01_第1章_<章节名>.md` |
| F2 | `02_第2章_<章节名>.md` |
| F3 | `03_第3章_<章节名>.md` |
| F4 | `04_第4章_<章节名>.md` |

### 工作流（强约束：每一处改动都要人介入）

降重不是 AI 全自动任务。AI 自动批量重写极易跑偏、脱离原意、丢失论证细节。固定流程：

1. **解析报告**：读取查重报告，按章节抽取每一段 U（直接命中）/Q（已引证仍重合）标记，记录相似字数、类型与原文关键短句。
2. **核查报告对位**：把每段 U/Q 片段在论文当前文件中定位到具体行号；同时检查是否存在大段照搬却未标注的情况。
3. **给出修改意见**（不直接动手）：按本节「输出格式」（见 §8）整理评估表，列出每一处的等级、改前/改后建议、命中报告段、相似字数。
4. **人工审核**：用户对着评估表逐条确认接受 / 修改 / 拒绝。
5. **在新上下文中单独应用每一处改动**：每次只打开与该处改动相关的引文与论文片段，单独执行编辑；上下文越纯净，越不会顺手改坏其他段落。

### 核心原则

- **原始内容是唯一标准**：降重只能改句式，不能改原意，更不能改到面目全非。任何看起来"更顺、更学术"但偏离原作者论证细节的改写都要回退。
- **只动 U/Q 标记句**：不在标记范围内的句子原则上不碰，避免无意义改动稀释原文风格。
- **改前必读引文**：执行某处改动的 AI 必须先把对应原始引文读一遍（仓库 `nlm/`、`references/`、`mineru_output/` 等），积累语感、记住原作者的论述方式与关键术语，确保改写后仍忠于引文含义；引文找不到时不动这一处，而是先去搜或问用户。
- **补引优先于改写**：若发现是大段照搬别人的话却未标脚注，先按 §4 流程补脚注/引证（这能直接把一段 U 转为 Q 的引证段，并不再计入"去除引用复制比"），再考虑要不要改写。
- **文献来源**：脚注文献先在仓库现有 md/参考文献列表中找；找不到再联网搜，搜不到则标"存疑文献"交用户判断。

### "无意义修改"判据

下列情形视为无效改动，应在评估表里标 N 级并回退：

- 改动位置完全落在 U/Q 标记区之外，对降重无帮助
- 改动只是换近义词或调整语气，但原句本就不在重复段内
- 已是正常引用（Q 段且脚注完整）的句子被无谓重写

### 评估等级

按"是否值得改 + 改的优先级"分级：

- **A 级**：相似 ≥100 字、U 类、改得对症 → 必须改（核查改得到不到位）
- **B 级**：相似 50–100 字 → 应改（评估改的质量）
- **C 级**：相似 <50 字 → 可改可不改
- **N 级（无效）**：改动不在任何 U/Q 段附近 → 应回退
- **缺漏**：报告里 ≥80 字的 U 段，但当前未做任何修改 → 必须补改

## 6. 文献格式规范

完整规范见 `assets/GBT7714-2015.md`；半角/全角分辨不清或怀疑 md 转录有误时回查 `assets/GBT7714-2015.pdf`。

### 项目级细则

GB/T 7714-2015 在不少地方留有解释空间。**每个论文项目首次启动时，与用户确认下列决策并固化在项目根目录的 `THESIS-RULES.md` 中**，后续修订统一参照。常见项：

下表"GB/T 依据"列引用的是 **GB/T 7714-2015 标准内部条款号**，不是本 SKILL.md 的章节号。

| 决策项 | 说明 | GB/T 7714 依据 |
|------|------|------|
| 引用日期统一值 | 所有在线文献的"引用日期 [YYYY-MM-DD]"是否统一为同一天 | GB/T §8.5 |
| 书名本体的年份/副标题 | `…选编(1978—2014)` 等年份是否照录题名页（不改冒号、不改括号） | GB/T §6.1 |
| 政府文件文号格式 | `(教社政〔XXXX〕X号)` 等是否在 `[A/OL]` 后用冒号分隔 | GB/T §8.2.3 |
| 报刊署名规则 | 领导人讲话/转述类是否署记者名；记者调查/观点类是否保留 | GB/T §3.2 |
| 多卷本写法 | `第1-4卷` 用"`:第1—4卷`"；中文数字"第一卷"统一改阿拉伯数字 | GB/T §8.2.3、§6.2 |
| `[Z]` vs `[EB/OL]` | 早于官方电子化年份的政策文件是否保留纸本类型标识 `[Z]` | GB/T §7 文献类型标识 |
| 著者人数缩略 | ≥4 人时是否统一"前 3 + 等"或全列 | GB/T §4.1.3 |

## 7. 多版本管理

**强提醒：检测到多版本时必须主动使用当前环境可用的提问能力向用户确认，禁止靠"最新修改时间""目录名最大日期"等推断自动选定。**

进入任何修改类任务前的固定动作：

1. **扫一遍项目根目录**，识别论文版本目录/文件，特征如：
   - 同名目录加日期/时间后缀（`<论文名>4.12晚/`、`<论文名>4.13下午/`）
   - 同名文件加版本号（`vN_章节名.md`、`xx_最终版.md`、`xx_盲审版.md`）
   - 同一章节出现于多处
2. **若发现 ≥2 个候选版本**：列出所有候选（带最近修改时间），使用当前环境可用的提问能力让用户明确指定要改哪一个，**得到答复前不动任何文件**。
3. **只有 1 个版本时**：可以直接进入工作，但首次回复中提一句"识别到唯一版本 XXX，将基于此修改"，给用户一次纠正机会。

**询问模板示例**：

```
检测到以下候选版本：
- 改革开放…4.12晚/   （最近修改：2026-04-12 22:31）
- 改革开放…4.13下午/  （最近修改：2026-04-13 17:51）
- 改革开放…4.13下午_v2/（最近修改：2026-04-14 09:00）
请问本次修改针对哪个版本？
```

项目内的 `.md`/`.txt` 文件可能既包含论文正文，也包含参考文献原文（来自 OCR、PDF 转录、`nlm/`、`mineru_output/` 等）；动手前先与用户对齐目录约定，**避免把文献原文当作论文正文修改**。

## 8. 核查输出格式

### 引用核查（单条）

```
------------------------------------------------------------
<作者/文献简称> —— <已修改 / 建议修改，待确认>

论文位置：
- 正文：第 N 行 / 第 X 章 X.X 节

文献引用：
- <完整 GB/T 7714-2015 格式>

原文来源：
- 文件：<项目内文献原文相对路径>
- 关键位置：第 N-M 行

原文依据：
- 第 N 行："<原文关键句>"

你原来的说法（修改前）：
- "<论文中的表述>"

问题在哪里：
- <问题描述>

我现在改成了 / 建议改为：
- "<新表述>"

结论：
- <一句话总结>
------------------------------------------------------------
```

确定的修改直接执行；不确定的列表标注"建议修改，待确认"，由用户决定。

### 降重评估表

降重场景下汇总输出的固定表头：

```
| 等级 | 文件 | 当前行 | 原文片段 | 改前→改后(简+依据) | 命中报告段 | 相似字数 | U/Q | 评价 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
```

字段说明：
- **等级**：A / B / C / N / 缺漏（定义见 §5）
- **文件**：F0–F4（约定见 §5）
- **当前行**：当前 working tree 中的行号区间，便于用户跳转
- **原文片段**：报告标记区内的关键短句（≤20 字），便于用户在不打开文件时也能快速对位
- **改前→改后(简+依据)**：两部分。① ≤30 字摘要，区分"删除 / 句式重写 / 换词 / 补脚注"；② 紧跟一行写改动原因或依据，例如「依据：XX 文献原文用'演变'而非'失效'」「依据：避免与 F2 第 88 行重复表述」「依据：报告段 7 共 133 字 U 直接命中」。原因不写或只写"降重"视为信息不足。
- **命中报告段**：如「第2章上·段7」，对应 `full.md` 里的 `### 段 N`
- **相似字数**：报告中"此处有 X 字相似"的 X，纯数字
- **U/Q**：报告标记类型（定义见 §5 开头「色标 → 字母约定」）
- **评价**：一句话说明改得对症 / 不到位 / 应回退 / 待补

排序：A → B → C → N → 缺漏。

输出后同样**只给意见、不动文件**；用户确认后由新上下文中的 AI 逐条应用（见 §5 工作流）。

### 重复说法清单（§3.2 用）

```
| 组号 | 类型 | 关键短语/文献 | 出现位置 | 上下文摘要 | 初判 |
| --- | --- | --- | --- | --- | --- |
```

字段说明：
- **组号**：G1, G2, …，便于后续逐组在新上下文中处理
- **类型**：关键词反复 / 同义反复段 / 跨章节同源引用
- **关键短语/文献**：被重复使用的短语，或被反复引用的文献简称
- **出现位置**：F1:120-122; F3:45-46; F4:88-90 这种格式，列全
- **上下文摘要**：每处 ≤20 字，方便用户一眼区分各处侧重
- **初判**：冗余可合并 / 应差异化保留 / 仅删某一处 / 待用户判断

排序：按"明显冗余 → 应差异化 → 待判断"递减。

每组用户确认后，单独开新上下文处理这一组（流程见 §3.2）。
