## 何时使用交互

### 必须交互的场景

| 场景 | 原因 | 示例 |
|------|------|------|
| 关键决策点 | 避免走错方向 | 技术方案选择 |
| 需要用户输入 | 无法自动获取 | API Key、文件路径 |
| 不可逆操作前 | 防止误操作 | 删除文件、覆盖数据 |
| 多种有效方案 | 尊重用户偏好 | 输出格式选择 |
| 阶段完成确认 | 确保方向正确 | 蓝图确认、结果验证 |

### 不需要交互的场景

| 场景 | 原因 |
|------|------|
| 有明确最佳实践 | 直接执行即可 |
| 可自动推断 | 从上下文获取 |
| 可撤销的操作 | 错了可以改 |
| 技术细节 | 用户不需要关心 |

## 问题设计原则

### 1. 问题要具体明确

问题要让用户一眼看懂在问什么、选完会发生什么。避免"你想怎么配置？"这种开放式问法——用户不知道你在问哪一层。改成"Skill 作用域是项目级还是全局级？"这种指向明确的问题。

### 2. 选项要互斥且完整

选项之间不能有重叠（比如同时出现"Markdown"和"文本格式"），也不能漏掉常见情况。是否需要手动加"其他"兜底项，取决于当前提问工具——有的会自动提供自由输入框，手动加反而重复；没有的就要自己兜。动手前确认一下所在环境的行为，再决定要不要加。

### 3. 选项数量适中（2-4 个）

- 2 个：非黑即白的二选一
- 3 个：主流路径 + 次选
- 4 个：上限，再多就拆成多步

超过 4 个往往说明分类粒度有问题，先重新归类或者分阶段问。

### 4. 提供足够上下文

利用已知信息让选项更有针对性。比如已经检测到项目用了 React，就在推荐项里注明"与现有项目一致"，而不是让用户在一堆框架里空选。选项标签里可以带简短理由，描述里给完整说明。

### 5. 推荐选项放第一位

有明确倾向时，把推荐项放第一个，并在 label 末尾标注 "(推荐)"。没有倾向就按常见度或字母顺序排，不要假装客观。

## 常用交互场景

按场景归类，每类说清楚**什么时候问、问什么方向、典型选项怎么切**。

- **需求澄清**：Skill 刚启动、用户表述模糊时问。方向是"核心功能属于哪一类"——文件处理 / 外部服务 / 领域工作流 / 其他。
- **技术方案**：同一目标有多条实现路径时问。每个选项用一句话讲清取舍（简单 vs 灵活、性能 vs 可读等）。
- **配置选择**：涉及作用域、路径、环境等硬约束时问。项目级还是全局级、本地还是远程、开发还是生产。
- **阶段确认**：蓝图敲定、第一版产出后问。通常给"符合继续 / 需要调整 / 重做"三档。
- **资源需求（多选）**：创建 Skill 时问要不要 scripts、references、assets。此时 `multiSelect` 开启。
- **测试选择**：测试阶段让用户挑场景——正常路径、边缘情况、或者跳过。
- **结果反馈**：每次产出后收集满意度，用于决定继续、调整还是重来。
- **异常处理**：报错或外部依赖不可用时问——跳过、换方案、还是停下等修复。

## 交互流程设计

### 渐进式询问

不要一次性把所有问题堆给用户。按**核心需求 → 技术方案 → 配置细节 → 执行确认**的顺序推进，每一步只问这一步必要的决策，让用户始终只面对一件事。

### 分支询问

后续问题依赖前一个答案时，不要提前问。比如先确定 Skill 类型是"文件处理"还是"API 集成"，再据此问"具体哪种文件格式"或"哪个 API"，避免问一堆用不上的东西。

### 确认循环

关键节点产出 → 展示结果 → 确认。满意就推进，不满意就回到上一步改，不要一路闷头走到底再让用户否决。

## 避免的反模式

1. **过度询问**：每个小决定都问。只在关键决策点问，其它直接按最佳实践执行。
2. **模糊选项**："选项 1 / 选项 2" 这种占位标签。每个选项都要讲清楚含义和后果。
3. **盲目加"其他"兜底**：先看当前提问工具是否已经自动提供自由输入框，再决定要不要手动加，避免重复。
4. **技术术语堆砌**：直接甩 "REST vs GraphQL with Apollo Client"。先用一句人话解释，再给选项。
5. **假设用户懂技术**：上来就问实现细节。先铺背景，再给选项，必要时提供"帮我决定"档。

## 小白友好原则

1. **解释术语**：技术词汇后跟一句人话说明。
2. **给出推荐**：不确定时明确告诉用户推荐哪个、为什么。
3. **提供示例**：抽象概念配具体例子。
4. **允许不知道**：提供"帮我决定"或"按推荐来"这类选项。
5. **渐进深入**：从简单问题开始，逐步往细节走，不要第一个问题就最难。
