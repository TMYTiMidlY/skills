# 质量检测

## 带宽测试 (iperf3)

目标机器上先启动服务端：

```bash
iperf3 -s
```

客户端测试：

```bash
iperf3 -c <目标IP>       # 测试上传
iperf3 -c <目标IP> -R    # 测试下载
iperf3 -c <目标IP> -P 4  # 并行流
```

## 延迟测试 (mtr)

```bash
sudo mtr -r -c 10 <目标IP> # 测试 10 次延迟
```

## IP / 网络检测

### IP 地址与 WebRTC 泄漏

访问 <https://browserleaks.com/webrtc>，检查：

- WebRTC 是否泄漏本地 / 公网真实 IP
- RTCPeerConnection、RTCDataChannel 支持情况

### DNS 泄漏

访问 <https://browserleaks.com/dns>，检查 DNS 请求是否走预期的出口。

### IP 风险检测

访问 <https://ping0.cc>，检查：

- IP 是否被识别为家宽 / 机房 IP
- IP 风险值评分
- 黑名单检测
- 端口开放情况

### 多节点延迟测试

访问 <https://www.itdog.cn>，支持从国内各地区、各运营商发起 Ping / Tcping / 路由追踪 / HTTP(S) 测速，适合评估 VPS 到国内的连通性。

## Aliyun 未备案封锁实测

2026-04 在阿里云大陆 ECS 上的实测观察：

- HTTPS 封锁看的是 TLS ClientHello 里的 SNI（明文），即目标域名。只要请求里带了未备案的域名，就会被拦。
- 最初用 `https://<域名>:8443` 访问是正常的；经过多次测试之后，家宽访问同一地址返回了阿里云"要求备案"的提示页，手机流量访问仍然正常。是不是反复测试导致命中的不确定，反正结果是家宽这边被拦了。

### 当前可用方案：IP 直连 + Caddy internal CA

结论：只要请求里不带域名就没事。浏览器用纯 IP 访问时默认不发 SNI，也没有域名可匹配，实测稳定可用，是目前在用的方案。

一个服务一个端口，`https://<主 IP>` 走默认的 443 也能正常访问；其他服务用各自的端口。TLS 统一 `tls internal` 自签，客户端装 Caddy local root CA，见 [extra-setup.md 的对应小节](extra-setup.md#安装-caddy-local-root-ca-tls-internal-场景)。

> 严格讲这是违规操作，V2EX 上有反馈过"一个星期后接到电话"的案例，共识是小规模用通常被容忍。长期要合规还是得备案或走海外前端反代。
